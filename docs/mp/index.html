Gem scan

Hindsight upper bound, not forward lift. This is the event-first agent finding the gem in a realistic, noisy GDELT news firehose — with its early under-the-radar article seeded (real search misses those niche pieces). It still leans on that seed and on a model trained past the events, so it does not prove the firehose finds gems in time. The clean verdict is the forward eval (src/forward.py).

The curated portfolio: each week, the gems the financial press names as thesis-driven movers go on the watchlist; a position is held while its driving thesis is live and dropped when it decays. A plain mean-variance optimizer sizes it — the LLM never sets a weight. Scan parameters

Plot 1 — Portfolio value

Plot 2 — Allocation over time

Capital committed per ticker (cash fills the rest). Fully invested while the watchlist is non-empty; to cash when the press names nothing live.

Plot 3 — Holdings timeline (proposed vs funded)

One row per ticker the curator named. Thin gray, small dots = proposed (on the live watchlist); thick colored, large dots = funded (the optimizer actually bought it). Gray-only rows are names the press flagged but the sizing floor pruned — what Plot 2/4 (funded only) don't show.

Plot 4 — Dollars held per ticker

Capital in dollars per ticker over time (cash fills to the portfolio total, so the stack's top edge is the portfolio value). Plot 2 shows the same split as percentages.

Plot 5 — Cumulative $ gain per holding

Total dollar P&L each holding contributed over the window (Σ daily position-value × daily return). Green = winner, red = loser; the bars sum to the portfolio's total gain.

Plot 6 — Watchlist by date

Each row is a date the live watchlist (or its funding) changed — the names the press kept thesis-live that week. Bold + colored = actually funded by the optimizer; gray = on the watchlist but pruned by the sizing floor.
